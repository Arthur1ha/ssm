# 智慧空间交互体验趋势调研报告

> 面向 HCI 毕设：多智能体赋能的室内位置感知智慧空间体验设计
> 撰写日期：2026-05-20

---

## 一、交互范式的演进脉络

智慧空间的人机交互方式经历了三代演进，目前正处于第二代向第三代的过渡阶段：

| 代际 | 交互模式 | 典型形态 | 认知负担承担方 |
|------|---------|---------|--------------|
| 第一代（2010s） | **显式命令控制** | App 滑块、语音开关灯 | 用户自行操作每个设备 |
| 第二代（2018–今） | **规则/场景自动化** | "回家模式""观影模式"、If-This-Then-That | 用户预先配置规则，执行时系统代劳 |
| 第三代（2023–今，前沿） | **意图驱动 + 主动感知** | 用户表达目标，系统理解并动态组合 | 系统承担能力发现与任务分解 |

**核心趋势关键词**：Ambient Intelligence / Anticipatory Computing / Intent-Based Interaction / Proactive Assistance / Context-Aware Automation

## 二、现有主流交互方式分类及局限

### 2.1 显式控制（第一代）

**代表**：米家 App、HomeKit、SmartThings 设备面板。

**体验特征**：
- 用户需要知道设备在哪里、叫什么名字、能做什么
- 一次操作对应一个设备
- 控制精确，但认知成本高

**局限**：
- 设备越多，入口越分散，学习成本线性上升
- 无法处理"我想做某件事"这类目标性表达
- 多设备联动需用户手动构思和配置

### 2.2 规则/场景自动化（第二代）

**代表**：Apple 捷径、Amazon Routines、米家自动化、Matter Automations。

**体验特征**：
- 预设"如果 X，则执行 [动作序列]"
- 用户一次配置，长期生效
- 场景粒度比单设备大（同时控制灯光+音乐+温度）

**局限**：
- **场景固定**：只能响应规则里预设的条件，无法动态适配用户当下真实状态
- **能力不可见**：用户仍需提前知道设备能做什么，才能配置规则
- **执行过程黑箱**：触发后用户不知道哪些步骤执行了、哪些失败了、为什么
- **规则爆炸**：随设备数量增加，规则维护成本指数级上升

### 2.3 自然语言 + 单 LLM（第二代升级版，2023–今）

**代表**：Google Gemini for Home、Amazon Alexa+LLM、Siri with Apple Intelligence。

**体验特征**：
- 用户说"把客厅调成适合看电影的氛围"，LLM 解析意图并下发多条设备指令
- 不再需要用户知道具体设备名称或操作步骤

**局限**：
- **仍然设备中心**：LLM 把用户请求翻译成设备指令列表，本质是"智能命令生成器"，没有"当前空间能做什么"的概念
- **执行过程不透明**：用户看不到系统是怎么决策的，无法干预
- **单点失败**：所有推理由一个 LLM 完成，无专业化分工
- **无法处理无效指令**：HomeBench（ACL 2025）研究发现，现有 LLM 在用户发出矛盾指令或不可执行命令时，成功率显著下降
- **不感知位置**：无论用户在哪个房间，系统响应逻辑相同

## 三、研究前沿概览

### 3.1 LLM 驱动的智能家居控制

目前被验证最有效的架构是 **Planner-Executor-Verifier** 模式，与 SSM 的 Planner→Dispatcher→Evaluator→Responder 高度对应：

- **SAGE（2023，arXiv:2311.00772）**：50 个复杂任务基准上达到 76% 成功率，关键创新是将用户请求分解为离散动作序列，每步有 grounding 约束防止幻觉
- **IoTGPT（2025）**：部署在 Samsung SmartThings，通过子任务记忆复用将延迟降低 78.4%、成本降低 44%
- **HomeBench（ACL 2025）**：首个评测 LLM 处理无效/矛盾指令的基准，发现这是现有系统最主要的失败场景

### 3.2 多智能体系统（MAS）在智慧空间中的应用

| 方案 | 优势 | 局限 |
|------|------|------|
| 规则引擎 | 确定性、低延迟、可解释 | 无法处理未预见场景 |
| 单 LLM Agent | 灵活、语义理解强 | 上下文窗口限制，单点失败 |
| 多智能体系统 | 职责分离、可并行、故障隔离、专业化 | 协调开销，通信协议设计复杂 |

主流架构分两类：**层次化**（中央协调器向专业子 Agent 委托）和**平等协作**（Agent 间直接协商，无中心节点）。2025 年新趋势是 MCP/A2A 协议取代传统 MQTT，允许 Agent 跨系统边界协作（ABI Research 预测）。

代表研究：
- *A Multi-Agent System for Service Provisioning in IoT Smart Space Based on User Preferences*（Sensors 2024）：提出动态服务发现与个性化匹配的多智能体编排模型
- *Multi-Agent-Based Smart-Home Energy Management with Adaptive Reasoning*（Applied Sciences 2026）：在关键负载条件下将能耗降低 16%

### 3.3 可解释性与用户控制感研究

**核心矛盾**：自动化程度越高 → 认知负担越低，但自主感和控制感也越弱。

关键结论（来自 CHI 2024 研究）：
- **"适度自动化"优于"完全自动化"**：28 名参与者实验发现，尽管全自动是最高效选项，大多数用户仍选择半自动，因为自我决定理论（SDT）中的自主需求——人需要感觉是自己在控制生活环境
- **"纠错能力"比"解释"更重要**：用户容忍度取决于能否轻松撤销/修改错误决策，而非解释有多详尽
- **不透明自适应行为**会显著提升隐私焦虑和控制感缺失

相关文献：
- *Connecting Home: Human-Centric Setup Automation in the Augmented Smart Home*（CHI 2024）
- *Better to Ask Than Assume: Proactive Voice Assistants' Communication Strategies*（CHI 2024）
- *AI Ethics in Smart Homes: Progress, User Requirements and Challenges*（arXiv:2412.09813）

### 3.4 室内位置感知技术现状

| 技术 | 精度 | 部署成本 | 适用场景 |
|------|------|---------|---------|
| BLE RSSI 指纹 | 1–3 m | 低 | 区域/房间级，智能家居最常用 |
| WiFi CSI 指纹 + 深度学习 | 1–2 m | 极低（复用路由器） | 2024–2025 年最活跃研究方向 |
| UWB | 10–30 cm | 高 | 厘米级精度，Apple U1/U2 |
| GPS 地理围栏 | 5–15 m（室外） | 零 | 建筑/楼层粒度区分 |

**对本研究的启示**：严格室内定位需要额外硬件（BLE beacon 或 UWB），实现成本较高。GPS 地理围栏可区分粗粒度空间（不同房子/不同楼层），文献中有大量支撑，是毕设原型的可行方案。"设备无关定位"（Device-Free Localization）通过分析 WiFi/BLE 信号变化感知人员存在，无需用户携带设备，是未来研究方向。

## 四、"多智能体方式"与现有交互方式的本质差异

### 4.1 现有分析的问题——差异表达不够准确

目前论文将研究差异定位为"多智能体 vs 传统显式设备控制"，但这个对比有两个问题：

**问题一**：Google Gemini for Home 已经能做自然语言意图控制，如果只说"不用自己开每个灯"，工业界已经实现了，论文贡献就显得薄弱。

**问题二**："传统显式控制"是非常宽泛的概念，智能家居的规则自动化（第二代）也不是"显式控制"，对比对象需要更精确。

### 4.2 更准确的差异表达框架

建议把差异从三个维度来表达：

#### 维度一：谁来承担"空间能力发现"的认知工作

| 方式 | 能力发现由谁完成 |
|------|----------------|
| 显式控制（第一代） | 用户自己探索每个设备的功能 |
| 规则自动化（第二代） | 用户在配置规则时预先完成 |
| 单 LLM 控制（第二代升级） | LLM 根据用户语言猜测意图，但仍以设备为中心 |
| **多智能体（本研究）** | **系统实时发现当前空间中可用的能力组合，向用户呈现** |

关键词：**从"设备中心"到"空间能力中心"**（Device-Centric → Space Capability-Centric）

#### 维度二：执行过程是否透明可干预

| 方式 | 执行过程透明度 |
|------|--------------|
| 显式控制 | 完全透明，但每步都需用户手动完成 |
| 规则自动化 | 黑箱自动执行，用户看不到过程 |
| 单 LLM 控制 | 黑箱输出结果，无中间步骤 |
| **多智能体（本研究）** | **计划生成→每步可确认→执行可中断→结果可解释** |

关键词：**可解释可干预的协作执行过程**（Transparent & Interruptible Coordination）

#### 维度三：位置如何参与服务组织

| 方式 | 位置的角色 |
|------|-----------|
| 显式控制 | 不感知位置 |
| 规则自动化 | 位置作为触发条件（进入房间开灯），静态规则 |
| 单 LLM 控制 | 基本不感知位置（全局统一处理） |
| **多智能体（本研究）** | **位置作为能力发现的入口——用户在哪里，系统就聚合那里的设备能力** |

关键词：**位置作为空间服务的动态组织入口**（Location as Capability Discovery Entry）

### 4.3 建议的核心贡献表述

> 现有智慧空间交互方式（包括规则自动化和 LLM 单智能体控制）仍以**设备**为基本操作单元：系统要么等待用户命令具体设备，要么将用户语言翻译为设备指令序列，但两者都无法向用户呈现"当前空间整体能做什么"，也无法让用户参与和理解多设备协调的过程。
>
> 本研究的核心贡献在于：通过多智能体协同架构，将交互对象从**设备**提升到**空间能力组合**，并通过透明可干预的执行过程，让用户既能以目标表达需求，又能在系统响应时保持对空间的掌控感。

## 五、SSM 原型的竞争力分析

### 与工业界的差异

| 特性 | Google Gemini Home | 米家AI | SSM（本研究原型） |
|------|-------------------|--------|-----------------|
| 自然语言理解 | 强 | 中 | 中（依赖 LLM API） |
| 多设备协调 | 有（单 LLM 生成命令列表） | 有（场景模式） | 有（多节点真实 MQTT 执行） |
| 执行过程透明 | 无 | 无 | **可实现（核心设计点）** |
| 执行步骤可干预 | 无 | 无 | **可实现（核心设计点）** |
| 空间能力摘要 | 无 | 无 | **可实现（核心设计点）** |
| 位置感知设备发现 | 品牌生态内可用 | 有 | **GPS 雷达已实现，可扩展** |
| 开放架构/可研究 | 封闭 | 封闭 | **MQTT 开放，可记录完整交互日志** |
| 硬件门槛 | 需购买生态设备 | 需购买生态设备 | **ESP32 ≈ ¥30，可自定义扩展** |

**关键洞察**：SSM 最大的竞争优势不是功能强，而是**开放可研究**——完整的多智能体协作日志、可自定义的设备能力组合、可调整的交互流程，这些是工业界封闭产品无法提供的研究条件。

### 与学术界的差异

SAGE、IoTGPT 等系统关注的是 **技术指标**（任务成功率、延迟）。本研究关注的是 **用户体验维度**（认知负担、掌控感、惊喜感），两者的研究问题正交，不构成直接竞争，反而构成引用关系。

## 六、对毕设的具体建议

### 建议一：修改论文核心对比维度

不要直接比较"多智能体 vs 传统显式控制"，而要比较：

> **"空间能力中心 + 透明协调"（B 组）vs "设备中心 + 黑箱执行"（A 组）**

A 组的对照条件：一个简单 PWA，列出所有设备，每个设备有独立的控制滑块，用户自己决定操作什么。这个对照更干净，也更接近现实中用户的使用体验。

### 建议二：用研量表选择

基于上述研究，建议 A/B 测试使用：
- **NASA-TLX**：测认知负担（Mental Load、Frustration 分项）
- **感知掌控感量表**（自编或参考 SDT 量表中 Autonomy 维度）：区分"感觉是自己在控制"和"感觉系统替我做决定"
- **半结构访谈**（Think-aloud 后）：挖掘"惊喜感"和"信任感"的质性描述

10 人做上述设计，可以支撑丰富的定性分析 + 有方向性的量化数据。

### 建议三：室内定位的处理策略

建议在论文中将位置感知分两层表述：
- **当前实现**：GPS 地理围栏（室外 / 粗粒度空间区分），已在 PWA 中实现
- **可扩展方向**：BLE RSSI 指纹（房间级，ESP32 硬件支持），作为 Future Work

这样既诚实，又留有扩展空间，避免答辩时被质疑"室内 GPS 不准"。

### 建议四：原型必须实现的两个功能

根据本文的差异分析框架，论文的核心论点依赖以下两个功能的实现，缺一不可：

1. **空间能力摘要**：用户进入某空间后，系统主动告知"这里可以帮你做什么"（由 LLM 根据 manifest 生成自然语言描述）
2. **执行过程可视化面板**：用户发出请求后，PWA 展示 Planner 的计划、Dispatcher 的执行步骤、Evaluator 的确认结果，并允许用户在任意步骤干预
